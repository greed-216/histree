# Histree 内容模型设计

## 为什么不能只靠“人物”和“事件”

如果 Histree 只把人物和事件做成简短卡片，内容一定会显得薄。一个可用的历史条目，至少需要同时回答五类问题：

1. 这是谁 / 这是什么。
2. 其一生或全过程是怎样展开的。
3. 它处在什么制度、地理、家族、阵营和社会结构中。
4. 它与哪些人、事、机构和地区发生联系。
5. 这些说法分别依据什么材料。

所以更合理的抽象不是“多加几个文本字段”，而是：

- 以人物、事件作为核心实体；
- 围绕核心实体组织多个结构化维度；
- 为重要叙述增加出处与证据层。

## 人物条目的完整模型

一个完整的人物条目，至少应包含以下层次。

### 1. 基本身份

- 姓名
- 字、号、别名
- 所处时代 / 国家 / 政权
- 阵营或政治归属
- 籍贯
- 生卒时间范围
- 画像 / 代表图像
- 标签

### 2. 生平叙事

- 简介
- 长篇生平
- 主要人生阶段
- 关键转折点
- 历史评价

### 3. 关系维度

- 亲属关系
- 政治关系
- 师承与思想关系
- 外交与军事关系
- 提携、依附、竞争与对立关系

### 4. 结构维度

- 官职经历
- 品级与晋升路径
- 所属机构
- 活动地理范围

### 5. 证据维度

- 百科链接，便于快速定位
- 原始史料
- 工具书 / 参考书
- 现代研究
- 面向具体事实的 claim 与 citation

## 事件条目的完整模型

一个完整的事件条目，也需要类似的层次。

### 1. 基本身份

- 标题
- 时间范围
- 地点 / 影响区域
- 所处时代与国家背景
- 标签
- 代表图像

### 2. 事件叙事

- 摘要
- 背景
- 过程分期
- 结果
- 历史影响

### 3. 参与者与阵营

- 关键参与人物
- 涉及国家 / 阵营
- 各参与者角色
- 前因后果事件

### 4. 结构维度

- 涉及行政区域
- 涉及官职与制度
- 军事、外交、财政等背景

### 5. 证据维度

- 百科链接
- 原始史料
- 工具书
- 现代研究

## 当前阶段的实现原则

Histree 现阶段不必一步到位，但应按下面的顺序推进：

1. 保持 `person` 与 `event` 为核心表。
2. 直接在两者上补足更丰富的描述字段和 `references`，先保证可用性。
3. 用 `source` 与 `fact_claim` 建立第一层证据体系。
4. 后续再拆分出官职经历、活动地点、时间跨度、出处挂载等更细的独立表。

## 当前产品的最低内容标准

每一个对外展示的人物或事件，至少应满足：

- 有可读的长文本概述或生平；
- 有标签；
- 有时代背景或阵营背景；
- 至少有一条参考资料链接；
- 对重要且不平凡的事实，至少要能追溯到一条明确出处。
